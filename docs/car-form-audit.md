# Аудит формы подачи объявления

## Текущее состояние

### Текущая форма

- Точка входа формы продажи: `app/listing/new/page.tsx`.
- Форма уже разбита на 5 шагов:
  - Паспорт
  - Техника
  - История и состояние
  - Фото и описание
  - Сделка и контакты
- Состояние формы хранится целиком на клиенте в одном `useState`, типы описаны в `lib/sale-form.ts`.
- Дублирование объявления подгружается через `GET /api/account/listings/[id]`.
- Сохранение нового объявления выполняется через `POST /api/listings`.

### Где сейчас лежат справочники и зависимости

- Марки и модели в форме берутся не из БД, а из статического файла `lib/car-catalog.ts`.
- Зависимость реализована только на уровне `марка -> модель` через `getModelsForMake`.
- Год, кузов, двигатель, привод, коробка, руль, цвет и комплектация сейчас выбираются из статических клиентских списков в `app/listing/new/page.tsx`.
- Регионы и города берутся из локального справочника `lib/ru-regions.ts` и `lib/ru-regions.generated.json`.

### Текущая БД

- ORM: Prisma, схема в `prisma/schema.prisma`.
- Основные таблицы предметной области:
  - `SaleListing`
  - `WantedListing`
  - `SellerProfile`
  - `ListingMedia`
  - `PriceHistory`
  - `Favorite`
  - `SavedSearch`
- Отдельного каталога автомобилей в БД нет.
- Данные авто сейчас лежат прямо в `SaleListing` в плоском виде:
  - `make`
  - `model`
  - `generation`
  - `year`
  - `bodyType`
  - `engine`
  - `engineDisplacementL`
  - `power`
  - `transmission`
  - `drive`
  - `steering`
  - `trim`
- В реальной базе сейчас отсутствуют таблицы справочника для марок, моделей, поколений, модификаций, двигателей, коробок и приводов.

### Текущие индексы в БД

- В `SaleListing` есть только общие индексы по публикации и поиску:
  - `(status, createdAt desc)`
  - `(status, publishedAt desc)`
  - `(createdByUserId, createdAt desc)`
  - `(status, make, year)`
  - `(status, city)`
  - `(status, price)`
- Эти индексы не подходят для зависимых селектов каталога.

## Основные проблемы

### 1. Каталог авто не нормализован

- Нет отдельных сущностей для бренда, модели, поколения, кузова, двигателя и модификации.
- Невозможно гарантировать логичность комбинаций, потому что форма собирает значения из отдельных строковых полей.
- Любое изменение верхнеуровневого поля не имеет надежной каталожной проверки для нижестоящих полей.

### 2. Клиент перегружен статическими справочниками

- `lib/car-catalog.ts` грузится в клиентский бандл.
- Форма зависит от большого статического списка вместо специализированной серверной выдачи.
- Справочники в коде и значения в БД живут отдельно и могут расходиться.

### 3. Зависимости между полями неполные

- Сейчас есть только `марка -> модель`.
- Нет зависимости:
  - `модель -> год`
  - `год -> кузов`
  - `кузов -> поколение`
  - `поколение -> двигатель`
  - `двигатель -> привод`
  - `привод -> коробка`
  - `комбинация -> модификация`

### 4. Нет канонического слоя данных

- В проект уже попали сторонние данные (`vehicles.csv`), но production-слой для staging/normalization отсутствует.
- Нет хранения источника происхождения, confidence-уровня, alias-ов и source mapping.

### 5. Совместимость со старыми объявлениями хрупкая

- Старые объявления живут только на raw string полях.
- Сейчас нет стратегии привязки существующих `make/model/generation/...` к каноническому каталогу.

### 6. Производительность будущих селектов сейчас не подготовлена

- Нет специализированных API под шаги формы.
- Нет индексов под фильтрацию по каталожным связям.
- Нет серверного кеширования каталожных запросов.

- `catalogModelId`

Это даст:

- возможность ручного fallback, если маппинг не найден.

## Как будет устроена новая схема данных

### Каноническая иерархия

- `brand`
  - `model`
    - `generation`
      - `modification`
        - `engine`
        - `drive`
        - `transmission`
        - `trim`
        - `body_type`

### Логика формы

- Марка запрашивается отдельным endpoint.
- После выбора марки сервер отдает только допустимые модели.
- После выбора модели сервер отдает только допустимые годы.
- После выбора года сервер отдает только кузова.
- После выбора кузова сервер отдает только поколения.
- После выбора поколения сервер отдает топливо, затем привод, затем коробку.
- После выбора комбинации сервер отдает список модификаций.
- После выбора модификации форма автоматически подставляет:
  - мощность
  - объем
  - человекочитаемое имя модификации
- Пользователь сохраняет возможность ручной корректировки значений там, где это допускается бизнес-логикой.

## Стратегия миграции без потери текущих объявлений

### Этап A. Добавить новый каталог рядом со старой моделью

- Создать новые таблицы каталога.
- Не удалять и не переименовывать существующие поля в `SaleListing`.
- Добавить nullable FK-поля каталога в `SaleListing`.

### Этап B. Загрузить каталог и включить API

- Загрузить данные сначала в staging.
- Выполнить нормализацию, дедупликацию и РФ-адаптацию.
- Залить канонические записи в production-таблицы каталога.
- Поднять отдельные API зависимых селектов.

### Этап C. Переключить форму на новый каталог

- Новая форма выбирает значения через каталог.
- При изменении поля верхнего уровня зависимые поля ниже сбрасываются.
- Старые raw-поля продолжают заполняться для совместимости.

### Этап D. Промапить старые объявления

- Написать backfill-скрипт, который пробует сопоставить старые `make/model/year/...` с новым каталогом.
- Где маппинг найден:
  - заполнить `catalog*Id`
- Где не найден:
  - оставить raw-поля как fallback
- Новое чтение и редактирование объявления должно работать и с mapped, и с raw-вариантом.

## Риски

- Внешние источники по-разному называют одну и ту же модель.
- У части источников нет точного понятия рестайлинга.
- У части источников есть поколение, но нет модификаций.
- У части источников есть модификации и моторы, но нет явной поколенческой структуры.
- Значит, нужен source-aware normalization layer, а не прямая загрузка в production.

## Решение по источникам данных

В проекте уже доступны и будут использоваться следующие источники:

- `vehicles.csv` из EPA как дополнительный источник powertrain- и variant-данных
- `vendor/vehicle-data/open-vehicle-db`
- `vendor/vehicle-data/automobile-models-and-specs`
- `vendor/vehicle-data/car-api`

План использования:

- `car-api`: бренд -> модель -> поколение -> годы
- `open-vehicle-db`: year/style/body coverage
- `automobile-models-and-specs`: engine specs, transmission, drive, power, displacement
- `vehicles.csv`: доп. powertrain/alias coverage, особенно для global market normalization

## Итог аудита

Текущий проект уже содержит пригодный каркас формы, объявления и общую бизнес-логику публикации. Основная проблема не во flow формы как таковом, а в отсутствии нормализованного автомобильного каталога и специализированного серверного слоя зависимых селектов. Без этого дальнейшее усложнение формы приведет к росту ручных исключений и битых комбинаций.

Следующий шаг: внедрить отдельный каталожный слой, импорт и API, а затем перевести на него шаги "Паспорт" и "Техника" без поломки `SaleListing`.
